Lessons Learned

팀 회고 — 잘된 점 / 문제점 / 개선사항

✓ 잘된 점 — 다음에도 유지할 것
Sprint Review를 통해 고객 피드백을 조기에 반영하여 요구사항 변경 발생 빈도가 줄었습니다.
→ 수행방식 합의에 Sprint Review 운영 기준 명시
✓ 잘된 점
본수행 착수점검에서 조건부 착수 항목을 사전에 식별하여 착수 후 블로커 발생이 감소했습니다.
→ Sprint 0 본수행 착수점검 절차 유지
✗ 문제점 — 반복되면 안 되는 것
데이터 제공 지연이 개발 Sprint 일정에 영향을 주었으나 RAID에 늦게 등록되었습니다.
→ 고객 준비도 체크리스트에 데이터 제공 일정 확인 항목 강화
✗ 문제점
요구사항 원문 출처가 일부 누락되어 검수 시 근거 확인에 시간이 소요되었습니다.
→ Requirement Source 필드 필수화
△ 개선사항 — Playbook 반영 권장
산출물 승인 기한이 Deliverable Matrix에 없어 승인 지연이 반복되었습니다.
→ Deliverable Acceptance Matrix에 승인기한 컬럼 추가
△ 개선사항
운영팀의 프로젝트 참여가 T-1주에 시작되어 인수인계 준비 시간이 부족했습니다.
→ 운영전환 Review를 T-4주로 앞당김

Loop

교훈 → 다음 Playbook 반영 지도

Lessons Learned는 보고서에만 남기지 않고 실제 Playbook에 반영되어야 합니다.

Sprint 0 Playbook 반영
고객 준비도 체크리스트 — 데이터 제공 일정 확인 강화

A2 책임 정렬 — 운영팀 참여 시점 T-4주 명시

Registry 마법사 — 원문 출처 기록 단계 강화
Deliverable Matrix — 승인기한 컬럼 추가
요구사항 상태 — 고객확인중 기준 명확화
수행 Play 반영
Sprint Review — 고객 피드백 기록 양식 추가
데이터 Play — 제공 일정 Milestone 필수화

Close Play — 운영전환 Review T-4주 시작 명시
Loop 실행 원칙: Lessons Learned는 PMO 또는 Playbook 관리자가 각 항목의 반영 여부를 다음 프로젝트 Sprint 0 체크리스트로 옮겨야 합니다. "보고서에 기록"으로 끝나면 Loop는 작동하지 않습니다.
