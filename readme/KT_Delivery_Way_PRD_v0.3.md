## 1. 제품 개요

### 1.1 제품명

**KT Delivery Way Platform**  
대안 명칭:

- KT AXD Project Playbook Service
- KT Delivery Playbook Agent
- KT AI/Agent Delivery Operating System

### 1.2 제품 정의

KT Delivery Way Platform은 **AXD 기반 AI/Agent 개발 수행 방법론**, **KT 내부 프로젝트 행정 프로세스**, **수행PM의 ERP/KOS/TM/VDC 단계별 운영 활동**을 하나의 프로젝트 수행 체계로 통합하여, 수행PM과 관리자가 프로젝트를 실행·점검·보고·관제할 수 있도록 지원하는 GPT Agent 기반 서비스이다.

서비스는 다음 세 축을 하나의 플레이북으로 통합한다.

| 축 | 설명 |
|---|---|
| AXD 수행 방법론 | AI/Agent 프로젝트를 Discovery, 설계, 구축, 실험, 평가, 검증, 전환, 운영 단계로 수행하기 위한 개발·수행 방법론 |
| KT 행정 프로세스 | 착수 PRB, 진행 PRB, 종료 PRB, 프로젝트 기본계획, 인력투입계획, 공정계획, 공정실적, Risk, 계약변경, 매출인식공정율, 하도급, 무상유지보수, 준공, 사후관리 등 KT 내부 관리 절차 |
| 수행PM 운영 활동 | 착수 전, 착수 시, 중간 보고, 완료 보고, 종료 시점별 KOS, ERP/PMS, TM, VDC, 수행원가, 직접비, 프로젝트 인센티브, 종료 PRB 업무 |

### 1.3 핵심 가치

기존 AXE Delivery Framework 또는 AXD 방법론이 프로젝트 수행 활동 중심이었다면, KT Delivery Way Platform은 프로젝트의 **수행 상태**와 **행정 처리 상태**를 동시에 관리한다.

```text
KT Delivery Way = AXD 개발방법론 + KT 행정 Control + 수행PM ERP/KOS/TM 활동 + Gate/PRB + GPT Agent 실행 지원
```

### 1.4 제품이 해결하려는 문제

KT의 AI/Agent 프로젝트 수행 과정에서는 다음 문제가 발생할 수 있다.

1. 개발 방법론과 KT 내부 행정 절차가 분리되어 관리된다.
2. 수행PM은 산출물, PRB, 공정, Risk, 계약, 하도급, 종료 처리를 개별 시스템과 문서로 관리해야 한다.
3. 관리자는 여러 프로젝트의 실제 위험 상태를 수행 관점과 행정 관점에서 통합적으로 보기 어렵다.
4. 프로젝트 단계 이동 시 AXD 산출물은 준비되었지만 KT 행정 Gate가 누락되는 상황이 발생할 수 있다.
5. PRB 준비자료, 주간보고, 리스크 대응안, 종료보고서 등 반복 산출물 작성에 시간이 많이 소요된다.
6. 프로젝트별 수행 방식과 행정 처리 기준이 사람의 경험에 의존하여 표준화가 어렵다.
7. 수행PM이 착수 전부터 종료까지 KOS, ERP/PMS, TM, VDC, 프로젝트 인센티브, 종료 PRB 관련 업무를 시점별로 놓치지 않고 처리해야 하지만, 현재는 단계별 체크리스트와 시스템 간 의존관계가 명확히 통합되어 있지 않다.

### 1.5 제품 목표

KT Delivery Way Platform의 목표는 다음과 같다.

- AXD 수행 방법론과 KT 행정 프로세스를 하나의 표준 생명주기로 통합한다.
- 프로젝트별 KT Delivery Way 플레이북을 자동 생성한다.
- 수행 활동, 산출물, 행정 Control, 수행PM ERP/KOS/TM/VDC 업무, PRB, Gate, Risk, 공정, 계약, 하도급, 준공 상태를 한 화면에서 관리한다.
- GPT Agent가 수행PM에게 다음 액션, 누락 항목, PRB 준비자료, 보고서 초안, 산출물 초안을 제안한다.
- 관리자가 여러 프로젝트의 수행 상태와 행정 처리 상태를 통합 관제할 수 있도록 한다.
- 주요 확정 업무는 사람 승인 기반으로 처리하여 통제성과 감사 가능성을 확보한다.

## 2. 참조 자료 및 전제

### 2.1 참조 자료

| 자료 | 활용 방식 |
|---|---|
| AXD 개발방법론 PDF | AI/Agent 프로젝트 수행 방법론, Phase, Activity, Task, 산출물, 역할, 품질 점검 기준의 기준 자료 |
| KT 프로젝트 관리 업무 프로세스 이미지 | KT 내부 프로젝트 행정 처리, PRB, 공정, Risk, 계약, 하도급, 준공, 사후관리 절차의 기준 자료 |
| 수행PM ERP/KOS/TM 단계별 업무 이미지 | 착수 전, 착수 시, 중간 보고, 완료 보고, 종료 시점에 수행PM이 KOS, ERP/PMS, TM, VDC, 수행원가/직접비, PRB에서 수행해야 하는 세부 업무의 기준 자료 |
| 기존 AXE Delivery Framework | 외부 Reference Framework로만 활용. 최종 사용자 화면과 서비스 용어는 KT 기준으로 재정의 |

### 2.2 주요 전제

- AXE Delivery Framework는 그대로 사용하지 않고, KT Delivery Way의 참조 모델로만 사용한다.
- 사용자 화면, Agent 응답, 단계명, 산출물명, Gate 기준은 KT 용어를 우선 적용한다.
- GPT Agent는 확정 처리를 직접 수행하지 않는다. PRB 확정은 타 시스템에서 처리하며, 계약변경 확정, 매출인식공정율 확정, 하도급 대가지급, 프로젝트 종료 확정 등 통제성 높은 업무는 사람 승인 후 수행한다.
- 초기 MVP는 외부 시스템 자동 처리보다 **조회, 누락 점검, 초안 생성, 다음 액션 추천, 승인 요청**을 중심으로 한다.
- 수행PM의 ERP/KOS/TM/VDC 단계별 활동은 단순 참고자료가 아니라 프로젝트별 Action Item과 Gate 보조 조건으로 관리한다.
- **PRB 요청, 검토, 회의, 심의결과 입력, 보완 요청 등록은 타 시스템 또는 기존 PRB 프로세스에서 처리한다.** 본 서비스는 PRB 항목을 단계와 Gate에 표시하되, 직접 처리하지 않고 처리 여부, 결과 상태, 보완 필요 여부, 외부 시스템 링크 또는 증적만 확인한다.
- PRB 준비자료 초안 생성은 수행PM 지원 범위에 포함하지만, PRB 등록·확정·심의결과 입력은 본 서비스의 처리 범위가 아니다.

| PRB 처리 시스템/타 PRB 프로세스 | 착수/진행/종료/사후관리 PRB의 요청, 검토, 회의, 심의결과 입력, 보완 요청은 타 시스템에서 처리한다. 본 서비스는 해당 처리 상태와 결과만 참조한다. |
| 계약 담당자 | 수주계약변경 요청과 확정 상태를 수행 프로젝트와 연동하여 관리하고 싶다. |
| 개발/AI/데이터/QA 담당자 | AXD 단계별 수행 활동, 담당 산출물, 테스트/평가/보안 점검 기준을 확인하고 싶다. |
| 관리자/PMO | 전체 프로젝트의 수행 방법론 준수 여부와 KT 행정 처리 준수 여부를 관제하고 싶다. |

### 3.2 주요 Persona

#### Persona A. 수행PM

- 프로젝트 착수부터 종료까지 전 과정을 관리한다.
- AXD 수행 산출물, KT 행정 절차, KOS/ERP/PMS/TM/VDC 단계별 업무를 함께 책임진다.
- 가장 자주 사용하는 기능:
  - 프로젝트 플레이북
  - 단계별 체크리스트
  - PRB 준비자료 초안 생성
  - 수행PM ERP/KOS/TM 단계별 할일 체크리스트
  - KOS 공조요청/직원 탭/계약 문서 상태 관리
  - VDC-A/B/C 검토 항목 관리
  - 프로젝트 인센티브 및 투입인력 TM 관리
  - 종료 PRB 타 시스템 처리 상태 및 필수항목 체크리스트
  - 주간보고 생성
  - Gate 통과 여부 점검
  - 다음 액션 추천

#### Persona B. 팀장/관리자

- 여러 프로젝트를 동시에 관리한다.
- 프로젝트 위험을 조기에 파악하고 조치를 지시한다.
- 가장 자주 사용하는 기능:
  - Portfolio Dashboard
  - 위험 프로젝트 브리핑
  - PRB/공정/Risk 누락 현황
  - 프로젝트별 Gate 상태
  - Agent 기반 관리자 요약

#### 참고 영역 C. PRB 처리 상태

- PRB 담당자의 실무 업무인 PRB 요청 접수, 검토, 회의 운영, 심의결과 입력, 보완 요청 등록은 타 시스템 또는 기존 PRB 프로세스에서 처리한다.
- KT Delivery Way Platform은 PRB 담당자용 처리 화면을 제공하지 않는다.
- 본 서비스에서는 PRB를 각 단계의 참고 Gate 항목으로 표시하고, 수행PM과 관리자가 처리 여부를 확인할 수 있도록 지원한다.
- 제공 기능:
  - PRB 대상 여부 표시
  - 타 시스템 PRB 처리 상태 조회 또는 수동 확인
  - 심의결과, 반려, 보완 요청 존재 여부 확인
  - 타 PRB 시스템 링크 또는 증적 파일 연결
  - Gate 영향도 표시
  - 수행PM 후속 조치 안내

## 4. KT Delivery Way Lifecycle

### 4.1 표준 단계

KT Delivery Way의 표준 생명주기는 다음 7단계로 정의한다.

```text
0. 사업/발주 및 착수준비
1. Discovery / 요구사항·AI 시나리오 정의
2. 설계·Sprint 계획
3. 구축·실험·통합
4. 평가·검증·보안점검
5. 전환·오픈·안정화
6. 준공·종료·사후관리
```

### 4.2 단계별 통합 매핑

| KT Delivery Way 단계 | AXD 수행 활동 | KT 행정 Control |
|---|---|---|
| 0. 사업/발주 및 착수준비 | 프로젝트 Context 정리, 역할 정의, 수행계획 준비 | 발주처리, 프로젝트 기본계획 관리, 착수 PRB 처리 상태 확인, 인력투입계획 등록, 공정계획 등록, 프로젝트 Risk 등록, 최초 하도급 계약 |
| 1. Discovery / 요구사항·AI 시나리오 정의 | 현행시스템 리뷰, 유저스토리 매핑, Agent AI 활용 시나리오 정의, Agent AI 서비스 정의, Agent AI 모델·정책 명세, 평가전략 수립, 요구사항 추적, Discovery 단계 점검 | 공정실적 등록, Risk 변경/대응 관리, 결과 점검·모니터링, 진행 PRB 대상 여부 확인 |
| 2. 설계·Sprint 계획 | Agent 아키텍처 설계, 평가기준 정의, PI 및 Sprint 계획 수립, DataOps 설계, AgentOps 설계, 서비스 관리도구 설계 | 진행 PRB 대상 여부 및 타 시스템 처리 상태 확인, 진행 PRB 결과 확인, 수주계약변경 필요 여부 검토 |
| 3. 구축·실험·통합 | DataOps 구현/테스트, AgentOps 구현/테스트, 서비스 관리도구 구현/테스트, Sprint 개발, 통합 준비 | 공정실적 등록, Risk 대응, 수주계약변경, 하도급계약 변경/추가신규/대가지급, 매출인식공정율 계획/등록 |
| 4. 평가·검증·보안점검 | Agent AI 평가, 통합 초기 테스트, E2E 시스템 테스트, 보안 취약점 점검, 단계별 품질 점검 | 진행 PRB 결과 확인, Risk 조치 현황 확인, 공정실적 반영, 계약변경 영향 확인 |
| 5. 전환·오픈·안정화 | 사용자 교육, 인수 및 검증 테스트, 딜리버리 준비·전환 계획, 시스템 전환·오픈, 안정화 | 기업프로젝트 산출물 조회, 종료 PRB 준비, 무상유지보수 정보 준비 |
| 6. 준공·종료·사후관리 | 안정화 결과 정리, 운영 인수인계, Lessons Learned | 종료 PRB 처리 상태 확인, 사후관리 PRB 처리 상태 확인, 무상유지보수 정보 등록, 하도급사 평가, 산출물 최종 확인 |

### 4.3 이중 Gate 원칙

KT Delivery Way에서 단계 전환은 아래 조건을 모두 검토해야 한다.

```text
단계 Gate 통과 = AXD 수행 Gate 통과 + KT 행정 Gate 통과 + 수행PM Action Gate 통과 + 승인 Gate 통과
```

| Gate 유형 | 설명 |
|---|---|
| 수행 Gate | AXD Activity 완료 여부, 필수 산출물 작성 여부, 품질 점검 결과 |
| 행정 Gate | PRB 처리 여부 확인, 공정실적, Risk, 계약변경, 매출인식, 하도급, 무상유지보수 등 KT 행정 처리 상태 |
| 수행PM Action Gate | KOS, ERP/PMS, TM, VDC, 수행원가, 직접비, 프로젝트 인센티브, 종료 PRB 관련 수행PM 업무 완료/확인 상태 |
| 승인 Gate | 수행PM, 팀장/관리자, 계약 담당자, 고객 담당자 승인 상태 및 타 PRB 시스템상 결과 상태 |
| 시스템 Gate | Easy ERP, KOS-B2B, Dashboard, 산출물 저장소 등 관련 시스템 등록/조회 상태 |

### 4.4 수행PM 할일 단계 매핑

| 수행PM 시점 | 주요 시스템/영역 | 핵심 수행PM 업무 | KT Delivery Way 반영 |
|---|---|---|---|
| 착수 전 | KOS, ERP/PMS, TM, VDC-A/B/C | 공조요청 확인, 직원 탭 투입기간 입력, KOS 승인 후 PMS 연동 확인, TM 가능 여부 확인, VDC 리스크/계약/수행계획 검토 | 사업/발주 및 착수준비 Gate 보조 조건 |
| 착수 시 | KOS, ERP/PMS, TM, 수행원가, 직접비 | 계약서 날인 후 발주요청, 착수 PRB 타 시스템 처리 상태 확인, 매출/비용 인식 방법 반영, 투입인력 계획 작성, TM 월별 관리, 하도급계약 확인 | 착수 PRB 상태 확인 및 착수 Gate 필수/조건부 조건 |
| 중간 보고 | 진행 PRB, 프로젝트 인센티브, TM | 진행 PRB 발생 시 타 시스템 처리 상태 확인, 인센티브 내부문서 결재, 엑셀 프로젝트 등록, 투입인력 TM 관리 | 진행 Gate 및 관리자 모니터링 조건 |
| 완료 보고 | 진행 PRB, 프로젝트 인센티브, TM | 완료 전 진행 PRB 발생 여부와 타 시스템 처리 결과 확인, 인센티브 결재/등록/TM 정합성 확인 | 완료 보고 Gate 보조 조건 |
| 종료 | 종료 PRB, 하도급, 청구 | 종료 PRB 타 시스템 처리 상태 확인, 검수/검사 조서, 사업규모별 첨부문서, 하도급 정산, 고객 청구작업 영업대표 안내 | 종료/준공 Gate 필수/조건부 조건 |

## 6. 핵심 사용자 시나리오

### 6.1 신규 프로젝트 플레이북 생성

1. 수행PM 또는 영업대표가 프로젝트 기본정보를 입력한다.
2. 시스템은 프로젝트 유형, 계약/발주 상태, 수행 범위, AI/Agent 적용 여부를 확인한다.
3. 시스템은 해당 프로젝트에 맞는 KT Delivery Way 단계와 체크리스트를 생성한다.
4. Agent는 착수 단계에서 필요한 AXD 준비 항목과 KT 행정 Control을 함께 제안한다.
5. 수행PM은 생성된 플레이북을 검토하고 필요한 항목을 확정한다.

#### 예시 출력

```text
현재 프로젝트는 '사업/발주 및 착수준비' 단계입니다.

필수 수행 항목:
- 프로젝트 Context 정리
- 역할/RACI 정의
- 수행계획 초안 작성

필수 행정 항목:
- 발주처리 상태 확인
- 프로젝트 기본계획 관리
- 착수 PRB 타 시스템 처리 상태 확인
- 인력투입계획 등록
- 공정계획 등록
- 프로젝트 Risk 등록

다음 액션:
1. 착수 PRB 준비자료 초안 생성 및 타 시스템 처리 상태 확인
2. 인력투입계획 등록 대상자 확인
3. 초기 Risk 목록 등록
```

### 6.2 단계 이동 가능 여부 점검

1. 수행PM이 “다음 단계로 넘어가도 되는지 확인해줘”라고 요청한다.
2. Gate Checker가 수행 Gate와 행정 Gate를 동시에 점검한다.
3. Agent가 누락 항목과 담당자별 액션을 제안한다.
4. PM 또는 관리자가 승인 후 단계 변경을 수행한다.

#### 예시 출력

```text
현재 프로젝트는 '설계·Sprint 계획' 단계입니다.

수행 Gate:
- Agent 아키텍처 설계서: 완료
- 평가기준 정의서: 완료
- Sprint Backlog: 진행중
- DataOps 설계서: 미완료

행정 Gate:
- 진행 PRB 대상 여부: 대상
- 타 시스템 PRB 처리 상태: 심의결과 미확인
- 공정실적 등록: 최신화 필요
- Risk 대응현황: High Risk 1건 미조치

판단:
다음 단계인 '구축·실험·통합'으로 이동하기에는 조건부 상태입니다.

필요 조치:
1. DataOps 설계서 완료
2. 타 PRB 시스템에서 진행 PRB 심의결과 처리 여부 확인
3. 공정실적 최신화
4. High Risk 대응계획 등록
```

### 6.3 PRB 준비자료 초안 생성 및 타 시스템 처리 상태 확인

1. 수행PM이 PRB 유형을 선택한다.
2. 시스템은 프로젝트 정보, 현재 단계, 산출물, Risk, 공정, 변경 이슈를 수집한다.
3. Agent가 PRB 준비자료 초안을 생성한다.
4. 수행PM은 타 시스템 또는 기존 PRB 프로세스에서 PRB 요청, 검토, 회의, 심의결과 입력을 진행한다.
5. 본 서비스는 타 시스템 PRB 처리 상태, 결과, 보완 필요 여부, 증적 링크만 조회 또는 수동 확인 상태로 반영한다.

지원 PRB 유형:

- 착수 PRB
- 진행 PRB
- 종료 PRB
- 사후관리 PRB

### 6.4 관리자용 다중 프로젝트 관제

1. 관리자가 Portfolio Dashboard를 연다.
2. 시스템은 프로젝트별 수행 상태와 행정 상태를 통합 계산한다.
3. Agent가 위험 프로젝트와 조치 필요 항목을 요약한다.
4. 관리자는 Escalation 또는 담당자 조치를 지시한다.

#### 예시 브리핑

```text
이번 주 주의/위험 프로젝트는 3건입니다.

1. A 프로젝트
- 단계: 평가·검증·보안점검
- 수행 이슈: E2E 테스트 결과서 미완료
- 행정 이슈: 타 PRB 결과 확인 대기
- 조치: 금요일 전 타 PRB 결과 확인 및 공정실적 최신화 필요

2. B 프로젝트
- 단계: 구축·실험·통합
- 수행 이슈: AgentOps 구현 지연
- 행정 이슈: 하도급 대가지급 예정일 초과
- 조치: 수행PM과 하도급 담당자 확인 필요

3. C 프로젝트
- 단계: 준공·종료·사후관리
- 행정 이슈: 무상유지보수 정보 미등록, 하도급사 평가 미완료
- 조치: 종료 PRB 타 시스템 처리 완료 확인 전 필수 항목 정리 필요
```


### 6.5 수행PM ERP/KOS/TM 단계별 할일 점검

1. 수행PM은 프로젝트 단계 탭에서 `착수 전`, `착수 시`, `중간 보고`, `완료 보고`, `종료` 중 현재 시점을 선택한다.
2. 시스템은 해당 시점에 필요한 KOS, ERP/PMS, TM, VDC, 수행원가/직접비, PRB 타 시스템 처리 상태 확인 항목을 체크리스트로 표시한다.
3. Agent는 완료되지 않은 항목, 선행 조건, 승인 필요 항목, 관련 자료 링크를 함께 안내한다.
4. KOS 승인 완료, PMS 자동 연동, TM 반영, 착수 PRB 타 시스템 처리 상태 확인, 종료 PRB 타 시스템 처리 완료 확인 등 시스템 간 의존관계를 Gate 판단에 반영한다.
5. 수행PM은 각 항목을 `미대상`, `준비중`, `요청완료`, `승인대기`, `완료`, `확인필요` 상태로 관리한다.

#### 예시 출력

```text
현재 시점: 착수 시

KOS 확인 항목:
- 공조 재요청 여부 확인: 완료
- 계약서 날인 후 발주요청: 영업대표 진행 필요
- 계약관련 문서 다운로드: 완료
- 계약서 날인 후 착수 PRB 타 시스템 처리 상태: 미확인

ERP/PMS 확인 항목:
- 수지분석표 기반 매출/비용 인식 방법 확인: 확인필요
- 투입인력 계획 작성: 진행중
- 하도급계약: 대상 여부 확인필요

TM 확인 항목:
- 투입인력 TM 반영 확인: 미완료
- 월별 TM 관리 기준 설정: 미완료

Gate 판단:
착수 단계 진행은 가능하나, 타 PRB 시스템에서 착수 PRB 처리 완료가 확인되고 투입인력 TM 반영이 완료되어야 착수 Gate를 통과할 수 있습니다.
```

### 8.1 Agent 목록

| 하도급/협력사 Agent | 하도급 계약, 변경, 추가신규, 대가지급, 하도급사 평가 상태를 점검한다. |
| 종료/준공 Agent | 종료 PRB 타 시스템 처리 상태, 무상유지보수, 사후관리 PRB 상태, 종료보고, 산출물 최종 확인을 지원한다. |
| 수행PM 할일 Agent | 착수 전, 착수 시, 중간 보고, 완료 보고, 종료 시점별 KOS/ERP/PMS/TM/VDC/PRB 처리 상태 확인 항목을 점검하고 누락 항목과 다음 액션을 제안한다. |

### 8.2 Agent 기본 동작 원칙

- Agent는 KT 표준 용어를 우선 사용한다.
- 외부 방법론 용어는 내부 매핑용으로만 사용하고 사용자 화면에 불필요하게 노출하지 않는다.
- Agent는 판단 근거를 함께 제시해야 한다.
- Agent는 주요 확정 업무를 자동 실행하지 않는다.
- PRB 요청·검토·회의·심의결과 입력·보완 요청 등록은 타 시스템에서 처리한다. Agent는 상태 확인, 자료 초안, Gate 영향도, 후속 액션만 제공한다.
- Agent는 사람 승인 필요 여부를 명확히 표시해야 한다.
- Agent 응답은 구조화된 형식으로 반환해야 한다.

```text
상단 Summary
- 프로젝트명
- 고객/사업
- 현재 KT Delivery Way 단계
- 수행 상태
- 행정 상태
- Gate 결과
- 위험도
- 종료예정일

좌측 Navigation
- KT Delivery Way Lifecycle
- 단계별 Activity
- 단계별 Control

중앙 Workspace
- AXD 수행활동 체크리스트
- KT 행정 Control 체크리스트
- 필수 산출물
- Gate 조건
- 담당자별 To-do

- 현재 단계 요약
- 누락 항목
```

### 9.2 PRB 상태 확인/참고 화면

#### 목적

착수/진행/종료/사후관리 PRB의 대상 여부와 타 시스템 처리 상태를 표시하고, PRB 상태가 단계 Gate에 미치는 영향을 확인한다. PRB 요청, 검토, 회의 운영, 심의결과 입력, 보완 요청 등록은 타 시스템에서 처리한다.

#### 주요 구성

- PRB 유형
- PRB 대상 여부
- 타 시스템 PRB 처리 상태
- 외부 시스템 링크 또는 증적
- PRB 준비자료 초안
- 보완 요청 존재 여부
- 심의결과 조회/확인 상태
- Gate 영향도
- 관련 산출물
- 관련 Risk

### 9.3 관리자 Portfolio Dashboard

#### 목적

관리자가 여러 프로젝트의 수행·행정·위험 상태를 통합 관제한다.

#### 주요 KPI

- 전체 프로젝트 수
- 정상/주의/위험 프로젝트 수
- PRB 지연 건수
- 공정실적 미등록 건수
- Risk 미조치 건수
- 계약변경 검토 대상 건수

## 11. KT 행정 Control 및 수행PM ERP/KOS/TM 업무 Catalog 초안

| Dashboard | 팀장/관리자 모니터링 데이터 연계 | 조회/요약 |
| 산출물 저장소 | 산출물 등록/조회/품질 점검 | 조회/업로드 연계 |
| Jira/Azure DevOps | Backlog, Sprint, Issue, Test 상태 | 선택 연동 |
| Confluence/SharePoint | 산출물/회의록/템플릿 저장 | 선택 연동 |
| GitHub/GitLab | 개발 진행, PR, 배포 상태 | 선택 연동 |

### 13.2 연동 원칙

- 조회와 상태 반영을 우선한다.
- PRB 처리는 타 시스템을 원천 시스템으로 간주하고, 본 서비스는 상태 조회·외부 링크·증적 반영만 수행한다.
- 확정, 승인, 지급, 종료 등 통제성 높은 업무는 사람 승인 후 처리한다.
- Agent가 수행한 제안, 생성, 판단, 요청은 감사 로그로 남긴다.
- 외부 시스템에 쓰기 연동을 추가할 경우 권한과 승인 Workflow를 필수 적용한다.

## 20. Acceptance Criteria 초안

### 20.1 프로젝트 플레이북

- Given 신규 프로젝트가 생성되었을 때  
  When 프로젝트 유형과 기본정보가 입력되면  
  Then 시스템은 KT Delivery Way 단계, 수행 Activity, 행정 Control, 필수 산출물, Gate 조건을 포함한 플레이북을 생성해야 한다.

### 20.2 Gate Checker

- Given 프로젝트가 특정 단계의 종료 시점에 있을 때  
  When 수행PM이 Gate 점검을 요청하면  
  Then 시스템은 AXD 수행 Gate와 KT 행정 Gate를 함께 점검하고, 통과/조건부/불가 상태와 근거를 표시해야 한다.

### 20.3 PRB 준비자료 초안 생성 및 상태 확인

- Given 프로젝트가 PRB 대상일 때  
  When 수행PM이 PRB 준비자료 생성을 요청하면  
  Then Agent는 프로젝트 개요, 현재 단계, 산출물, Risk, 공정, 계약/하도급 영향, 요청사항을 포함한 PRB 준비자료 초안을 생성해야 한다.

- Given 타 PRB 시스템에서 PRB가 처리되는 경우  
  When 수행PM 또는 연동 API가 PRB 상태를 확인하면  
  Then 시스템은 PRB 요청·검토·심의결과 입력을 직접 수행하지 않고 처리 상태, 심의결과, 보완 요청 여부, 외부 링크만 표시해야 한다.

### 20.4 관리자 관제

- Given 여러 프로젝트가 등록되어 있을 때  
  When 관리자가 Dashboard를 조회하면  
  Then 시스템은 프로젝트별 수행상태, 행정상태, Gate상태, 위험도, 조치 필요 항목을 표시해야 한다.

### 20.5 승인 통제

- Given PRB 확정, 계약변경 확정, 매출인식 등록, 하도급 대가지급, 종료 확정 작업이 요청되었을 때  
  When Agent가 해당 작업을 수행하려 하면  
  Then 시스템은 자동 실행하지 않고 승인 요청을 생성해야 한다.


### 20.6 수행PM 할일 Workbench

- Given 프로젝트가 착수 전 또는 착수 시점에 있을 때  
  When 수행PM이 `수행PM 할일` 화면을 조회하면  
  Then 시스템은 KOS 공조요청, 직원 탭 투입기간, KOS 승인 후 PMS 연동, TM 입력, VDC 검토, 착수 PRB 타 시스템 처리 상태 확인 필요 여부를 표시해야 한다.

- Given 프로젝트가 종료 단계에 있을 때  
  When 수행PM이 종료 체크리스트를 조회하면  
  Then 시스템은 종료 PRB 타 시스템 처리 상태, 검수/검사 조서, 사업규모별 첨부문서, 하도급 정산, 고객 청구작업 영업대표 안내 항목을 표시해야 한다.

- Given 프로젝트 인센티브 대상 프로젝트일 때  
  When 수행PM이 중간 보고 또는 완료 보고 단계 체크리스트를 조회하면  
  Then 시스템은 내부문서 결재, 엑셀 내 프로젝트 등록, 투입인력 TM관리 항목을 조건부 필수로 표시해야 한다.

## 21. 부록 A. 용어 정의

| 용어 | 정의 |
|---|---|
| KT Delivery Way | AXD 개발방법론과 KT 행정 프로세스를 통합한 KT형 프로젝트 수행 표준 |
| AXD | AI/Agent 프로젝트 수행을 위한 개발방법론. Phase, Activity, Task, 산출물, 역할 체계를 포함 |
| Control | KT 내부 행정 처리 항목. PRB 타 시스템 처리 상태 확인, 공정, Risk, 계약변경, 하도급, 무상유지보수 등 |
| Gate | 다음 단계로 이동하기 위한 수행/행정/승인 조건 |
| PRB | 프로젝트 착수, 진행, 종료, 사후관리 관련 심의·검토 절차. 본 서비스에서는 직접 처리하지 않고 타 시스템 처리 상태를 확인하는 참고 Gate로 사용 |
| 플레이북 | 프로젝트별 단계, Activity, Control, 산출물, Gate, 담당자, 다음 액션을 모은 실행 가이드 |
| 행정 상태 | KT 내부 시스템 또는 절차상 처리 상태 |
| 수행 상태 | AXD 활동, 산출물, 품질, 테스트, 전환 등 개발 수행 관점의 상태 |
| 수행PM 할일 | 착수 전부터 종료까지 수행PM이 KOS, ERP/PMS, TM, VDC 등에서 처리하거나 PRB 타 시스템 처리 상태를 확인해야 하는 시점별 업무 |
| KOS | 공조요청, 발주요청, 계약 관련 문서 확인 등 착수 전후 업무가 수행되는 시스템/업무 영역 |
| TM | 투입인력의 공수 입력, 반영, 월별 관리를 수행하는 업무 영역 |
| VDC-A/B/C | 발의서, 리스크 등급 평가, 통합과업대비표, 수지분석표, 법무검토본, 수행계획서 등 착수 전 검토 유형. 정확한 적용 기준은 추가 정의 필요 |
| 프로젝트 인센티브 | 대상 프로젝트에 대해 내부문서 결재, 엑셀 등록, 투입인력 TM관리 등을 수행해야 하는 조건부 관리 항목 |
