## 2. Readonly Playbook 원칙

본 Close Playbook은 시스템이 아니라 **프로젝트 종료 시 참고하는 Readonly 가이드**이다. 따라서 사용자가 데이터를 입력하거나 저장하는 기능, 보고서를 자동 생성하는 기능, 상태를 변경하는 기능은 포함하지 않는다.

| 원칙 | 설명 |
|---|---|
| Readonly | 사용자는 내용을 읽고 프로젝트 종료 활동에 참고한다. |
| 실행 가이드 중심 | 무엇을 확인하고, 무엇을 결정하고, 어떤 산출물을 남겨야 하는지 안내한다. |
| 입력 기능 없음 | 화면 내 Form, 저장, 수정, 삭제, 첨부, 승인 기능을 제공하지 않는다. |
| 상태관리 없음 | Activity 완료율, 지연, Open Item 상태 변경 등 동적 상태관리를 제공하지 않는다. |
| 자동 생성 없음 | 종료보고서, 체크리스트, Lessons Learned Report를 시스템이 자동 생성하지 않는다. |
| 산출물 템플릿 안내 | 필요한 산출물의 이름, 목적, 작성 기준은 안내하되 파일 생성 기능은 제공하지 않는다. |
| 개발 단순성 | 정적 HTML, Markdown, JSON 콘텐츠 데이터 기반으로 구현 가능해야 한다. |

## 3. Close Playbook 한 줄 정의

> **Close Playbook은 고객사 종료보고로 프로젝트를 공식 종료하고, Project Offboarding으로 안전하게 철수한 뒤, Lessons Learned로 교훈을 정리하고, 내부 종료보고와 프로젝트팀 해산식으로 프로젝트를 완결하는 Readonly 실행 가이드이다.**

---

## 4. 배경 및 문제 정의

프로젝트 종료 단계는 종료보고서 작성이나 검수 확인만으로 끝나지 않는다. 고객사 종료보고 이후에도 고객사 철수, 인력 철수, 권한 회수, 운영 및 하자보수 이관, 잔여사항 정리, Lessons Learned, 내부 종료보고, 프로젝트팀 해산식이 이어진다.

이 활동들이 명확한 가이드 없이 수행되면 다음 문제가 발생한다.

| 문제 | 설명 |
|---|---|
| 고객 종료와 내부 종료 혼재 | 고객사 종료보고와 내부 이해관계자 종료보고가 같은 메시지로 처리되어 목적이 불명확해진다. |
| 철수 활동 누락 | 인력 철수, 권한 회수, 자료 반납, 운영/하자보수 이관이 누락될 수 있다. |
| 잔여사항 관리 미흡 | 종료 후 처리해야 할 이슈, 결함, 보완 요청의 담당자와 기한이 불명확하다. |
| Lessons Learned 형식화 | 회고가 단순 의견 공유로 끝나고 잘된 점, 개선점, 재발 방지사항이 구조화되지 않는다. |
| 팀 해산의 비공식화 | 프로젝트팀 역할 종료가 명확히 선언되지 않아 후속 문의와 책임 경계가 모호해진다. |
| 초보 PM 실행 어려움 | 종료 단계에서 무엇을 질문하고 무엇을 확인해야 하는지 알기 어렵다. |

---

## 5. Close Playbook 목적

Close Playbook의 목적은 다음과 같다.

> **PM이 프로젝트 종료 단계에서 고객사 종료보고, Offboarding, Lessons Learned, 내부 종료보고 및 팀 해산식을 누락 없이 수행하도록 Activity 기반 Readonly 가이드를 제공하는 것**

이를 통해 프로젝트는 고객 기준으로 안정적으로 종료되고, 프로젝트팀은 안전하게 철수하며, 수행 교훈은 내부 이해관계자에게 공유되고, 프로젝트팀은 공식적으로 해산된다.

| 제외 항목 | 제외 사유 |

| 데이터 입력 Form | Readonly Playbook 목적에 맞지 않음 |
| 저장/수정/삭제 기능 | 시스템이 아니라 정적 가이드이므로 제외 |

| Activity 상태관리 | 진행중/완료/지연 등 상태값을 시스템이 관리하지 않음 |
| Open Item 관리 기능 | 미완료 항목 관리 방법은 안내하되 시스템 기능으로 제공하지 않음 |

| 결재 Workflow | 고객/내부 승인 프로세스는 각 조직 절차에 따름 |
| CSAT | 고객 만족도 조사는 이번 Close Playbook 범위에서 제외 |
| 후속 영업 Opportunity 관리 | 영업 프로세스 영역으로 분리 |
| 자산화 전용 프로세스 | 별도 자산관리 Playbook 또는 PMO 프로세스로 분리 |
| 다중 사용자 협업 | Readonly 정적 가이드 범위 초과 |

# 10. Close Playbook Overview

## 10.1 Close란

Close는 프로젝트 수행 결과를 고객 기준으로 정리하고, 프로젝트팀과 고객 환경을 안전하게 분리한 뒤, 수행 교훈을 남기고 내부적으로 프로젝트를 공식 마무리하는 단계이다.

Close는 단순 종료보고가 아니라 다음 네 가지 활동을 포함한다.

1. 고객사 기준 프로젝트 종료
2. 고객사 및 수행 환경에서의 Offboarding
3. Lessons Learned를 통한 교훈 정리
4. 내부 종료보고와 프로젝트팀 공식 해산

## 10.2 Close Playbook 목적

| 목적 | 설명 |
|---|---|
| 안정적 종료 | 계약 범위, 산출물, 검수, 잔여사항을 기준으로 고객사 종료를 공식화한다. |
| 안전한 철수 | 인력, 권한, 자료, 운영/하자보수 이관을 누락 없이 정리한다. |
| 교훈 정리 | 잘된 점, 개선점, 원인, 재발 방지사항을 구조화한다. |
| 내부 공유 | 고객 종료 결과, Offboarding 결과, Lessons Learned를 내부 이해관계자에게 공유한다. |
| 팀 공식 해산 | 프로젝트팀의 역할 종료와 기여 인정을 통해 공식적으로 팀을 해산한다. |

## 10.3 Close 한눈에 보기

| No | Activity | 요약 | 대표 산출물 |
|---|---|---|---|
| A1 | 프로젝트 종료 | 고객사 기준으로 프로젝트 종료보고를 수행하고 종료 상태를 확정한다. | 고객사 종료보고서, 종료보고 회의록, 고객 종료 판정 |
| A2 | Project Offboarding | 고객사 철수, 권한 회수, 자료 정리, 운영/하자보수 이관을 확인한다. | Offboarding Checklist, 권한 회수 목록, 잔여사항 이관 목록 |
| A3 | Lessons Learned | 이번 프로젝트의 잘된 점, 개선점, 원인, 재발 방지사항을 정리한다. | Lessons Learned Report, 재발 방지사항 정리표 |
| A4 | 프로젝트 팀해산식 | 내부 종료보고와 팀 기여 인정을 통해 프로젝트팀을 공식 해산한다. | 내부 종료보고 자료, 팀 해산식 Agenda, 해산 기록 |

## 10.4 Activity별 정리 결과가 활용되는 위치

| Activity | 정리 결과 | 다음 활용 위치 |
|---|---|---|
| A1 프로젝트 종료 | 고객 종료 결과, 검수 상태, 잔여사항 | A2 Offboarding, A4 내부 종료보고 |
| A2 Project Offboarding | 철수 상태, 권한 회수, 운영/하자보수 이관, 잔여사항 이관 | A3 Lessons Learned, A4 내부 종료보고 |
| A3 Lessons Learned | 잘된 점, 개선점, 원인, 재발 방지사항 | A4 내부 종료보고, 프로젝트 Archive |
| A4 프로젝트 팀해산식 | 내부 종료보고 결과, 팀 기여 인정, 해산 기록 | PMO 종료 기록, 프로젝트 Archive |

## 10.5 이 페이지 활용 방법

- PM은 A1부터 A4까지 순서대로 확인한다.
- 각 Activity에서 핵심 질문, 결정해야 할 것, 완료 기준을 먼저 확인한다.
- Task별 Play Guide를 참고하여 실제 회의, 보고, 점검 시 사용할 질문을 준비한다.
- 이행 확인 체크리스트는 실제 수행 후 누락 여부를 점검하는 참고표로 사용한다.
- 본 Playbook은 Readonly 가이드이므로, 실제 산출물 작성과 승인 기록은 프로젝트 문서 저장소 또는 조직의 공식 절차에 따른다.

# 11. A1. 프로젝트 종료

## 11.1 Activity 개요

A1 프로젝트 종료는 고객사 기준으로 프로젝트를 공식 종료하기 위한 Activity이다. PM은 계약 범위, 수행 결과, 최종 산출물, 검수 상태, 잔여사항을 정리하고 고객사 종료보고를 수행한다.

이 단계의 목적은 단순히 종료보고서를 작성하는 것이 아니라, 고객 기준으로 프로젝트가 종료 가능한 상태인지 확인하고 종료 또는 조건부 종료에 대한 합의를 기록하는 것이다.

## 11.2 핵심 질문

> **고객사 기준으로 프로젝트를 공식 종료해도 되는가?**

## 11.3 결정해야 할 것

| 구분 | 결정해야 할 내용 |
|---|---|
| 종료 기준 | 계약서, 제안서, SOW, 사업수행계획서, 검수 기준 중 종료 판단 기준 |
| 종료보고 범위 | 수행 결과, 산출물, 검수 상태, 잔여사항, 하자보수 전환 포함 여부 |
| 산출물 기준 | 필수 산출물, 최종 버전, 제출 방식, 승인 상태 |
| 검수 상태 | 검수 완료 / 조건부 검수 / 검수 대기 / 검수 미완료 |
| 잔여사항 처리 | 종료 전 해결 / 조건부 종료 / 운영 이관 / 하자보수 이관 |
| 고객 승인자 | 고객 PM, Sponsor, 검수 확인자 |
| 종료보고 방식 | 대면 보고 / 온라인 보고 / 문서 보고 / 회의록 대체 |
| 종료 판정 | 종료 가능 / 조건부 종료 / 종료 보류 |

## 11.4 완료 기준

| 완료 기준 | 확인 근거 |
|---|---|
| 고객사 종료보고 자료가 준비되었다. | Customer Close Report |
| 계약 범위와 수행 결과가 정리되었다. | Scope Closure Matrix |
| 최종 산출물 제출 상태가 확인되었다. | Final Deliverable List |
| 검수 또는 조건부 검수 상태가 명확하다. | Acceptance Evidence List |
| 잔여사항 처리 방식이 합의되었다. | Residual Item List |
| 고객사 종료보고가 수행되었다. | 종료보고 회의록 |
| 고객 종료 승인 또는 조건부 종료 합의가 기록되었다. | Customer Closure Decision |

# 12. A2. Project Offboarding

## 12.1 Activity 개요

A2 Project Offboarding은 고객사 종료보고 이후 프로젝트팀이 고객 환경과 프로젝트 수행 체계에서 안전하게 철수하기 위한 Activity이다.

이 단계는 단순 인력 철수가 아니라 운영/하자보수 이관, 권한 회수, 고객 자료 반납/폐기, 잔여사항 이관, 역할 종료를 포함한다.

## 12.2 핵심 질문

> **프로젝트팀이 고객사와 수행 환경에서 안전하게 철수할 준비가 되었는가?**

## 12.3 결정해야 할 것

| 구분 | 결정해야 할 내용 |
|---|---|
| 철수 범위 | 고객사 현장 철수, 원격 지원 종료, 협업도구 종료 범위 |
| 인력 철수 | 철수 대상자, 철수일, 후속 연락체계 |
| 운영 이관 | 운영 담당자, 인수 항목, 이관 완료 기준 |
| 하자보수 전환 | 하자보수 담당자, 접수 채널, 지원 기간, 처리 기준 |
| 권한 회수 | VPN, VDI, 시스템, Git, 협업도구, 저장소 권한 |
| 자료 정리 | 고객 자료 반납, 개인정보 삭제, 보안자료 폐기 |
| 잔여사항 이관 | 운영 이관 / 하자보수 / 고객 조치 / 내부 후속 조치 |

## 12.4 완료 기준

| 완료 기준 | 확인 근거 |
|---|---|
| 철수 대상 인력과 일정이 확정되었다. | Project Offboarding Plan |
| 운영 이관 항목이 확인되었다. | Operation Handover Checklist |
| 하자보수 전환 기준이 정리되었다. | Warranty Support Plan |
| 권한 회수 대상과 회수 상태가 정리되었다. | Access Revocation List |
| 고객 자료 반납/폐기 상태가 확인되었다. | Security Closure Checklist |
| 잔여사항 이관 대상이 명확하다. | Residual Item Transfer List |

| A4 프로젝트 팀해산식 | 팀 해산 전 역할 종료와 철수 완료 상태 공유 |
| 내부 종료보고 | Offboarding 완료 현황 보고 |
| 운영/하자보수 조직 | 잔여사항 및 운영 지원 항목 이관 |

# 13. A3. Lessons Learned

## 13.1 Activity 개요

A3 Lessons Learned는 이번 프로젝트 수행 과정에서 얻은 교훈을 정리하는 Activity이다.

이 단계에서는 다음 프로젝트 반영사항이나 방법론 개선까지 확장하지 않고, 이번 프로젝트에서 잘된 점, 개선점, 원인, 재발 방지사항, 내부 종료보고에 반영할 메시지를 정리하는 데 집중한다.

## 13.2 핵심 질문

> **이번 프로젝트에서 무엇을 잘했고, 무엇을 반복하지 말아야 하는가?**

## 13.3 결정해야 할 것

| 구분 | 결정해야 할 내용 |
|---|---|
| 회고 범위 | 전체 프로젝트 / 특정 단계 / 특정 이슈 / 종료 과정 |
| 회고 참석자 | PM, PL, QA, 주요 팀원, 운영 인수자, 필요 시 고객 PM |
| 회고 방식 | Workshop / 설문 / 인터뷰 / 회의록 기반 정리 |
| 교훈 분류 | 잘된 점 / 개선점 / 재발 방지사항 / 내부 공유 메시지 |
| 재발 방지 대상 | 일정, 범위, 품질, 의사소통, 인력, 기술, 보안, 고객 대응 |
| 내부 보고 반영 | 어떤 내용을 내부 이해관계자에게 공유할 것인가 |

## 13.4 완료 기준

| 완료 기준 | 확인 근거 |
|---|---|
| 회고 세션 또는 의견 수집이 수행되었다. | 회고 회의록, 설문 결과 |
| 잘된 점이 정리되었다. | What Went Well List |
| 개선점과 원인이 정리되었다. | Improvement Point List, Root Cause Summary |
| 재발 방지사항이 정리되었다. | Prevention Action List |
| 내부 종료보고에 반영할 메시지가 정리되었다. | Internal Close Message |

# 14. A4. 프로젝트 팀해산식

## 14.1 Activity 개요

A4 프로젝트 팀해산식은 프로젝트 수행 결과와 Lessons Learned를 내부 이해관계자에게 공유하고, 프로젝트팀의 역할 종료를 공식화하는 Activity이다.

이 단계는 단순 회식이나 이벤트가 아니라 내부 종료보고, 성과 공유, 교훈 공유, 팀 기여 인정, 공식 팀 해산 선언을 포함하는 Closure Ceremony이다.

## 14.2 핵심 질문

> **프로젝트 결과와 교훈을 내부에 공유하고, 프로젝트팀을 공식적으로 해산했는가?**

## 14.3 결정해야 할 것

| 구분 | 결정해야 할 내용 |
|---|---|
| 내부 종료보고 대상 | Sponsor, 본부장, PMO, 관련 리더, 후속 운영/지원 조직 |
| 보고 범위 | 고객 종료 결과, 수행 성과, Offboarding 결과, Lessons Learned |
| 해산식 방식 | 보고 회의 / 간담회 / 타운홀 / 소규모 팀 세션 |
| 참석자 | 프로젝트팀, 내부 이해관계자, 지원조직 |
| 팀 기여 인정 방식 | 감사 메시지, 우수 기여자 언급, 성과 공유 |
| 공식 해산 기준 | 역할 종료, 철수 완료, 내부 보고 완료, 잔여사항 이관 완료 |

## 14.4 완료 기준

| 완료 기준 | 확인 근거 |
|---|---|
| 내부 종료보고 자료가 준비되었다. | Internal Close Report |
| 고객 종료 결과가 내부에 공유되었다. | Customer Closure Summary |
| Offboarding 완료 상태가 공유되었다. | Offboarding Summary |
| Lessons Learned가 공유되었다. | Lessons Learned Summary |
| 팀 기여 인정과 공식 해산 선언이 수행되었다. | Closing Ceremony Record |
| 후속 잔여사항 담당 조직이 확인되었다. | Transfer List |

## 16.1 구현 원칙

| 접근성 | 제목 계층, 표, 링크가 명확해야 한다. |
| 유지보수성 | Activity와 Task 콘텐츠는 코드와 분리하여 관리할 수 있어야 한다. |

# 17. 성공 기준

| 성공 기준 | 판단 기준 |
|---|---|
| PM이 Close 흐름을 이해할 수 있음 | A1~A4 Activity 흐름과 목적을 설명할 수 있다. |
| 고객사 종료보고 준비에 도움이 됨 | 종료 기준, 산출물, 검수, 잔여사항 확인 항목을 찾을 수 있다. |
| Offboarding 누락을 줄일 수 있음 | 권한, 자료, 운영 이관, 하자보수 항목을 체크할 수 있다. |
| Lessons Learned를 구조화할 수 있음 | 잘된 점, 개선점, 원인, 재발 방지사항을 구분하여 정리할 수 있다. |
| 내부 종료보고와 팀 해산식 준비에 도움이 됨 | 고객 종료 결과, Offboarding, Lessons Learned, 팀 기여 인정 항목을 확인할 수 있다. |
| Readonly 목적을 위반하지 않음 | 입력, 저장, 자동생성, 상태관리 기능이 없다. |

# 18. 운영 원칙

| 원칙 | 설명 |
|---|---|
| 고객 종료와 내부 종료를 분리한다. | 고객사 종료보고와 내부 이해관계자 종료보고는 목적과 메시지가 다르다. |
| Offboarding은 사전 준비 후 실행한다. | 권한 회수, 운영 이관, 잔여사항 목록은 A1 단계에서 초안을 준비하고 A2에서 확정 실행한다. |
| Lessons Learned는 이번 프로젝트 교훈에 집중한다. | 다음 프로젝트 반영, 방법론 개선까지 확장하지 않는다. |
| 팀해산식은 공식 Closure Ceremony로 본다. | 단순 이벤트가 아니라 내부 종료보고와 역할 종료 선언을 포함한다. |
| 모든 잔여사항은 Owner와 Due Date를 가져야 한다. | 조건부 종료와 종료 보류를 구분하기 위한 필수 기준이다. |
| 기록되지 않은 종료는 완료로 보지 않는다. | 종료보고, 회의록, 체크리스트, 해산 기록 중 하나 이상의 근거가 있어야 한다. |

# 19. 최종 메시지

> **Close Playbook은 고객사 종료보고로 프로젝트 종료를 공식화하고, Project Offboarding으로 안전하게 철수한 뒤, Lessons Learned로 이번 프로젝트의 교훈을 남기고, 내부 종료보고와 프로젝트팀 해산식으로 프로젝트를 완결하는 Readonly 실행 가이드이다.**
