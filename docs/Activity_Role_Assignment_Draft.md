# Sprint 0 / Close Playbook — Activity별 참여 담당자(Role) 식별 (검토용 초안 v2)

> 이 문서는 화면에 바로 반영하지 않고, PM(작성자) 확인·수정 후 각 HTML Playbook에 반영하기 위한 **중간 산출물**입니다.
> v1(최초 식별) 대비 아래 통합 규칙을 적용해 역할 수를 단순화했습니다.

## 역할 통합 규칙 (확정)

| 기존 역할(v1) | 최종 역할(v2) |
|---|---|
| PM (수행PM) | **PM** |
| Scrum Master / Agile Lead | **PM** |
| 사업관리 담당 (PMO 성격) | **PM** |
| 운영/교육 담당 | **PM** |
| 인프라/환경 담당 | **클라우드/DevOps** |

| 운영/하자보수 조직 | **운영조직** |

## 0. 최종 역할(Role) 목록

| Role | 소속 | 정의 |
|---|---|---|
| **PM** | 수행사 | 프로젝트 총괄 책임자. Sprint 운영·Ceremony 진행, 회의체·보고서·문서관리, 오픈·전환·교육·운영전환 고려사항 확인까지 포함해 전 Activity의 실질적 진행·조율·최종 책임을 맡는다. |
| **PL / Tech Lead** | 수행사 | 기술/개발 수행 책임자. 기술 검토, 산출물 작성 책임, 환경·도구 목록 정의 |
| **QA** | 수행사 | 품질 기준, 검토 절차, 산출물 승인 기준, 위험평가. 내부 종료보고 자료 검토(PMO 역할 포함) |
| **클라우드/DevOps** | 수행사 | 환경·저장소·Branch·백업·배포 도구 정의, 접근권한·환경 리스크 확인 |
| **보안 담당자** | 수행사 또는 고객사 겸임 | 보안 정책, 접근권한, 개인정보, 자료반출 기준 |
| **데이터 담당자** | 수행사 또는 고객사 겸임 | 데이터 제공 일정, 마스킹 기준 확인 |
| **운영조직** | 수행사 내부(또는 별도 조직) | 운영 인수, 하자보수 접수·처리, 잔여사항 이관 대상 |
| **협력사 담당자** (조건부) | 협력사 | 협력사 참여 시 책임 경계 확인 |
| **고객 PM** | 고객사 | 고객 측 카운터파트, 승인·검토·의사결정 |
| **고객 Product Owner** | 고객사 | Backlog 우선순위, 검수 기준 결정 |
| **고객 IT/인프라 담당** | 고객사 | 계정 발급, 환경 준비, 접근 승인 |
| **Sponsor** | 고객사 | 최상위 의사결정, No-Go/조건부 승인 대상, 내부 종료보고 수신 대상 |

## 1. Sprint 0 Playbook — Activity별 담당자

### A1. 프로젝트 수행방식 합의 (수행방식 및 Sprint 운영 기준 정렬)
| Role | 역할 |
|---|---|
| PM | Workshop 주관, 수행방식 최종 승인, Sprint 운영 기준·작업 시작/완료 기준 초안 작성, 사업수행계획서 반영 |
| 고객 Product Owner | Backlog 우선순위, 검수 기준 검토 |
| PL / Tech Lead | 개발·기술 수행 가능성 검토 |
| QA | 품질 기준, 작업 완료 기준, 산출물 검토 기준 정렬 |
| 고객 PM | 고객 보고·승인 체계와 충돌 여부 확인 |

### A2. 수행팀 투입 및 R&R 정렬 (참여자 역할과 의사결정 구조 확인)
| Role | 역할 |
|---|---|
| PM | RACI 초안 작성, 책임 공백 식별, 고객 확인 진행 |
| 고객 PM | 고객 측 담당자·승인자·의사결정자 지정 |
| PL / Tech Lead | 기술 수행 책임자와 검토자 지정 |
| QA | 품질검토와 승인 흐름 확인 |
| 보안 담당자 | 보안 준비 책임 확인 |
| 클라우드/DevOps | 환경·인프라 준비 책임 확인 |
| (조건부) 협력사 담당자 | 협력사 참여 시 책임 경계 확인 |

### A3. 의사소통관리 및 보고·회의 주기 수립
| Role | 역할 |
|---|---|
| PM | 의사소통관리 체계 최종 책임, 회의체 운영계획, 주간/월간보고, 회의록 관리, Sprint Ceremony 운영과 일정 조율 |
| 고객 PM | 고객 보고·회의 주기 승인 |
| PL | 파트별 보고와 Action Item 처리 |
| QA | 품질점검회의와 시정조치 추적 기준 연계 |

### A4. 산출물 검토·승인 기준 정의
| Role | 역할 |
|---|---|
| PM | 산출물 목록과 제출 일정 통합 관리, 문서번호·저장위치·변경이력 관리 기준 반영 |
| QA | 산출물 품질 기준과 검토 절차 정의 |
| PL / Tech Lead | 개발·기술 산출물 작성 책임 확인 |
| 고객 PM | 고객 승인자와 검토 절차 확정 |
| 고객 Product Owner | 요구사항·Backlog 관련 산출물 승인 기준 검토 |

### A5. 환경·도구·접근권한 준비
| Role | 역할 |
|---|---|
| PL / Tech Lead | 필요 환경·도구·권한 목록 정의 |
| 고객 IT/인프라 담당 | 계정 발급, 환경 준비, 내부망 접근 승인 |
| 보안 담당자 | 권한 범위, 보안접속, 자료반출 기준 검토 |
| PM | 지연 리스크와 일정 영향 관리, 문서저장소와 산출물 보관 기준 반영 |
| 클라우드/DevOps | 저장소, Branch, 백업, 배포도구 기준 정의 |

### A6. 고객 측 준비 현황·보안성·리스크 점검
| Role | 역할 |
|---|---|
| PM | Readiness Review 주관, 위험·이슈·의사결정 통합 관리 |
| 고객 PM | 고객 조치 필요사항의 Owner 지정과 일정 확인 |
| QA | 위험평가 기준, 위험등급, 대응계획 검토 |
| 보안 담당자 | 보안 준비도, 개인정보, 자료반출 기준 검토 |
| 데이터 담당자 | 데이터 제공 일정과 마스킹 기준 확인 |
| 클라우드/DevOps | 환경·권한 준비 리스크 확인 |

### A7. Sprint 1 실행준비 및 미정 항목 이관
| Role | 역할 |
|---|---|
| PM | Day 1 Review 주관, 최종 착수 판단 제안, 오픈·교육·운영전환 고려사항 확인 |
| 고객 PM | 고객 측 착수 가능 여부 확인과 조건부 승인 |
| PL / Tech Lead | 기술·환경·작업 준비상태 확인 |
| QA | 품질 기준과 산출물 승인 기준 준비상태 확인 |

## 2. Close Playbook — Activity별 담당자

### A1. 프로젝트 종료 (Customer Closure)
| Role | 역할 |
|---|---|
| PM | 종료 기준 확인, 산출물/검수 상태 종합, 종료보고 주관, 종료 판정 기록(Customer Closure Decision) |
| 고객 PM | 종료보고 목차 확인, 검수 확인서 발급, 최종 승인자 서명 |
| PL | 산출물 목록/Repository 대조, 누락 산출물 식별 |
| Sponsor | 필요 시 고객 측 최종 승인권자 |

### A2. Project Offboarding
| Role | 역할 |
|---|---|
| PM | Offboarding 범위/일정 확정, 잔여사항 이관 대상·기한 지정 |
| PL | 운영 이관 시연, 인수 서명 수령 |
| 보안 담당자 | 권한 회수 확인, 자료·개인정보 삭제 증적 관리 |
| 클라우드/DevOps | 협업도구·시스템 접근권한·저장소 권한 회수 처리 |
| 운영조직 | 시스템 운영 인수, Runbook 수령, 하자보수 접수 채널·담당자 지정, 잔여 결함 처리 |

### A3. Lessons Learned
| Role | 역할 |
|---|---|
| PM | 회고 범위/방식 결정, Facts 정리, 핵심 메시지 선별(Internal Close Message) |
| PL | 팀 관점 회고 참여(기술/일정 이슈 원인 분석) |
| QA | 품질 관련 원인 분석 참여 |
| 팀 전체(참여자) | Workshop/설문/인터뷰 참여, 잘된 점·개선점 의견 제공 |

### A4. 프로젝트 팀해산식 (Closing Ceremony)
| Role | 역할 |
|---|---|
| PM | 내부 종료보고 대상/일정 확정, 자료 작성, 팀 기여 인정, 해산 선언(Project Team Dissolution Record) |
| Sponsor | 내부 종료보고 대상(수신자) |
| QA | 내부 종료보고 자료 검토, 프로젝트 종료 상태 관리 |
| 운영조직 | 잔여사항 이관 확인 대상 |
| 프로젝트팀 전원 | 해산식 참석 대상, 기여 인정 수혜자 |

## 3. 다음 단계

역할 매핑이 확정되었으므로, 다음으로 반영 방식을 정해야 합니다.

1. **Sprint 0**: 현재 `activity_data.json`에만 있고 화면(HTML)에는 owners 정보가 노출되지 않는 상태. A1~A7 각 화면에 위 표를 어떤 형태(요약 표 신설 vs 기존 문구 보강)로 반영할지
2. **Close**: 지금까지 담당자 정보가 전혀 없던 A1~A4 화면에 위 표를 신규 반영할 위치와 형태(Sprint 0와 동일한 스타일 권장)

방향을 알려주시면 실제 HTML 파일에 반영하겠습니다.
