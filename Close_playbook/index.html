Close Playbook 개요

프로젝트 종료 단계
Readonly 실행 가이드

Close는 프로젝트 수행 결과를 고객 기준으로 정리하고, 프로젝트팀과 고객 환경을 안전하게 분리한 뒤, 수행 교훈을 남기고 내부적으로 프로젝트를 공식 마무리하는 단계입니다.

고객사 기준 프로젝트 종료 Project Offboarding Lessons Learned 내부 종료보고·팀해산식
Readonly Playbook 원칙
본 Playbook은 시스템이 아니라 프로젝트 종료 시 참고하는 Readonly 가이드입니다.

사용자가 데이터를 입력하거나 저장하는 기능, 보고서를 자동 생성하는 기능, 상태를 변경하는 기능은 포함하지 않습니다. 무엇을 확인하고, 무엇을 결정하고, 어떤 산출물을 남겨야 하는지 안내합니다.

Four Activities

4개 Activity 개요

각 카드를 클릭하면 상세 가이드 페이지로 이동합니다.

A1
프로젝트 종료
고객사 기준으로 프로젝트 종료보고를 수행하고 종료 상태를 확정합니다.
고객사 기준으로 프로젝트를 공식 종료해도 되는가?
대표 산출물: 고객사 종료보고서, 종료보고 회의록, 고객 종료 판정
A2
Project Offboarding
고객사 철수, 권한 회수, 자료 정리, 운영/하자보수 이관을 확인합니다.
프로젝트팀이 안전하게 철수할 준비가 되었는가?
대표 산출물: Offboarding Checklist, 권한 회수 목록, 잔여사항 이관 목록
A3
Lessons Learned
이번 프로젝트의 잘된 점, 개선점, 원인, 재발 방지사항을 정리합니다.
무엇을 잘했고, 무엇을 반복하지 말아야 하는가?
대표 산출물: Lessons Learned Report, 재발 방지사항 정리표
A4
프로젝트 팀해산식
내부 종료보고와 팀 기여 인정을 통해 프로젝트팀을 공식 해산합니다.
결과와 교훈을 내부에 공유하고 팀을 공식 해산했는가?
대표 산출물: 내부 종료보고 자료, 팀 해산식 Agenda, 해산 기록

참여 Role

Close에 참여하는 역할과 담당 범위

Close Playbook A1~A4에는 아래 역할이 참여합니다. 각 Activity 개요에 표시된 "참여자"는 이 역할 정의를 기준으로 합니다.

수행사

  • PM — 프로젝트 총괄. 종료보고·Offboarding·회고·해산식 전 과정 주관, 최종 산출물·판정 기록 책임
  • PL / Tech Lead — 산출물 제출 상태 확인, 운영 이관 시연, 회고 시 기술·일정 이슈 원인 분석 참여
  • QA — 검수 상태 확인, 회고 시 품질 이슈 원인 분석, 내부 종료보고 자료 검토 및 종료 상태 관리
  • 클라우드/DevOps — 협업도구·시스템 접근권한·저장소 권한 회수 처리
  • 보안 담당자 — 권한 회수 확인, 고객 자료·개인정보 삭제 증적 관리
  • 운영조직 — 시스템 운영 인수, Runbook 수령, 하자보수 접수·처리, 잔여사항 이관 확인
  • 프로젝트팀 전원 — 회고 Workshop 참여, 해산식 참석 및 기여 인정 대상

고객사

  • 고객 PM — 종료보고 목차 확인, 검수 확인서 발급, 최종 승인자 서명
  • Sponsor — 고객 측 최종 승인권자, 내부 종료보고 수신 대상
이 페이지 활용 방법

Close Playbook은 이렇게 참고합니다

운영 원칙
  • PM은 A1부터 A4까지 순서대로 확인한다.
  • 각 Activity에서 핵심 질문과 결정해야 할 것을 먼저 확인한다.
  • Task별 Play Guide를 참고하여 실제 회의, 보고, 점검 시 사용할 질문을 준비한다.
  • 이행 확인 체크리스트는 실제 수행 후 누락 여부를 점검하는 참고표로 사용한다.
  • 본 Playbook은 Readonly 가이드이므로, 실제 산출물 작성과 승인 기록은 프로젝트 문서 저장소 또는 조직의 공식 절차에 따른다.
