설계 원칙

5가지 핵심 원칙

이 원칙을 지키지 않으면 요구사항 관리는 문서 작성으로 끝납니다.

원칙 1. 요구사항은 출처가 있어야 한다

모든 요구사항은 최소 하나 이상의 Source(RFP·제안서·기술협상서·인터뷰·회의록·현행분석)와 연결되어야 합니다. 출처 없는 요구사항은 등록할 수 없습니다.

원칙 2. 요구사항과 Backlog는 분리하되 연결한다

요구사항은 고객과 공식 합의하는 기준, Backlog는 수행팀이 실행하는 단위입니다. 둘은 다른 문서지만 RequirementBacklogMap으로 반드시 연결되어야 합니다.

원칙 3. 영역별 수행방식은 달라야 한다

기능 개발은 Sprint, 인프라·보안은 Gate, 데이터 이관은 Waterfall+Rehearsal, 인터페이스는 Hybrid입니다. 모든 요구를 Sprint로 관리하면 안 됩니다.

원칙 4. 요구사항은 테스트와 연결되어야 한다

요구사항이 테스트 케이스와 연결되지 않으면 검수 시 누락됩니다. Acceptance Criteria 없는 요구사항은 확정 요구사항이 아닙니다.

원칙 5. 변경은 요구사항·Backlog·WBS·테스트에 동시 반영되어야 한다

변경요청은 한 문서만 바꾸는 것이 아닙니다. 요구사항정의서 → Backlog → WBS → 설계 산출물 → 테스트 케이스까지 추적 관계 전체를 갱신해야 합니다.
