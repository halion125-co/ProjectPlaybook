Module 02 — 요구사항 분류·작성

요구사항을 올바르게 쓰고 분류하는 법

요구사항 작성 공식, AX 10개 영역별 분류 기준, PM 친화 Registry 입력 순서, 요구사항 상태값 체계를 제공합니다.

나쁜 예시 vs 좋은 예시
✗ 나쁜 요구사항
"고객정보 조회가 필요하다."
Actor 없음 권한 조건 없음 성능 기준 없음 수락 기준 없음
✓ 좋은 요구사항
영업관리자는 고객 상담 시 고객 기본정보·계약정보·문의이력을 한 화면에서 조회할 수 있어야 한다. 개인정보는 권한에 따라 마스킹 처리되며, 조회 결과는 3초 이내 표시되어야 한다.
작성 공식

요구사항 작성 6요소 공식

[Who]는 [Why]를 위해
[What] 대상에 대해
[Data] 데이터를 활용하여
[Action]을
[Condition/Acceptance] 조건·기준에 따라 수행할 수 있어야 한다.
예시: 영업관리자(Who)는 고객 상담 효율화(Why)를 위해 고객 계정(What) 대상에 대해 기본정보·계약정보·문의이력(Data)을 통합 조회(Action)할 수 있어야 하며, 개인정보는 마스킹 처리되고 응답시간은 3초 이내여야 한다(Condition/Acceptance).
기능 요구사항 템플릿
[사용자]는 [업무 목적]을 위해 [대상 데이터/업무]를 [조회/등록/수정/승인/처리]할 수 있어야 한다.
보안 요구사항 템플릿
시스템은 [보호 대상 정보]에 대해 [인증/권한/암호화/마스킹/로그]를 적용해야 한다.
AI/자동화 요구사항 템플릿
시스템은 [입력 데이터]를 기반으로 [추천/분류/예측/요약] 결과를 제공해야 하며, 결과는 [평가 기준]을 만족해야 한다.
인프라 요구사항 템플릿
프로젝트는 [개발/검증/운영] 환경에서 [시스템/서비스]를 수행할 수 있도록 [서버/계정/권한]을 준비해야 한다.
데이터 요구사항 템플릿
시스템은 [원천 데이터]를 [수집/정제/이관/검증]하여 [목표 시스템/업무]에서 사용할 수 있어야 한다.
운영 요구사항 템플릿
운영자는 [서비스/배치]의 상태를 모니터링하고, [장애/오류] 발생 시 [알림/조치/복구]를 수행할 수 있어야 한다.

Registry 입력

Registry 입력 마법사

이 순서대로 입력하면 요구사항이 완성됩니다.

1
이 요구는 어디서 나왔나요?
→ Source (RFP·제안서·기술협상서·인터뷰·회의록·현행분석)
2
고객이 말한 원문은 무엇인가요?
→ Original Statement
3
누가 원하는 요구인가요?
→ Requester / Actor
4
무엇을 하고 싶은 건가요?
→ Requirement Description
5
왜 필요한가요?
→ Business Purpose
6
어떤 데이터/시스템과 관련 있나요?
→ Data / System
7
기능인가요, 보안/데이터/인프라/운영인가요?
→ Requirement Type + Domain
8
누가 구현해야 하나요? (개발/패키지/인프라/보안)
→ Implementation Type
9
어떻게 완료 여부를 확인하나요?
→ Acceptance Criteria
10
어떻게 수행할까요? (Sprint/Waterfall/Kanban/Gate)
→ Execution Method
11
우선순위는 무엇인가요? (Must/Should/Could/Won't)
→ Priority
12
복잡도는 어느 정도인가요? (상/중/하)
→ Complexity
13
확정된 요구인가요, 확인이 필요한가요?
→ Status (분류완료 / 분석중 / 고객확인중)
완성도 점검: Source, Actor, Acceptance Criteria, 검증 방법, 수행 영역, 구현 방식 중 하나라도 비어 있으면 확정 요구사항이 아닙니다. "기준선 확정" 상태로 올리기 전에 6가지를 모두 채우세요.
