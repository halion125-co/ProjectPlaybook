Module 03 — 수행전략·추적

수행방식을 선택하고
Backlog·WBS·테스트로 연결한다

요구사항 영역별 수행방식 판정 기준, Backlog/WBS 변환 규칙, 요구사항추적표 구성, 변경관리 절차를 제공합니다.

5가지 수행방식 판정 기준 Backlog 변환 규칙 WBS 추가 정보 가이드 요구사항추적표 구성 변경관리 CR 절차
핵심 원칙
모든 요구를 Sprint로 관리하면 안 됩니다.
영역마다 맞는 수행방식이 다릅니다.
Sprint — 화면/업무기능 개발, 고객 피드백이 필요한 것
Gate — 보안·통합테스트·인수테스트, 단계별 승인 필요
Kanban — 인프라·권한, 상태 기반 흐름 관리
Waterfall — 데이터 이관, 순서와 검증이 중요
Hybrid — 인터페이스, 설계는 Gate + 구현은 Sprint

Kanban — 인프라·권한·운영 작업
Sprint 주기 없이 상태(To Do → In Progress → Done)로 관리하는 작업에 적합합니다.
Kanban 대상예시 작업상태값

Waterfall + Rehearsal — 데이터 이관
순서와 검증이 중요하고 한번 실패하면 되돌리기 어려운 작업에 적합합니다.
단계작업검증 방법

정제·변환데이터 정제, 변환 로직 구현샘플 데이터 검증
Rehearsal 1차전체 데이터 이관 리허설건수 비교, 오류율 확인

변경관리

요구사항 변경관리 CR 절차

기준선 확정 이후 변경은 반드시 CR 절차를 따릅니다.

1
변경 요청 등록
변경 요구자, 변경 내용, 변경 사유, 영향도(범위·일정·비용·품질)를 CR 양식에 작성
2
영향도 분석
PM, PL, QA가 요구사항→Backlog→WBS→산출물→테스트 전체 영향 분석
3
CCB 심의
변경관리위원회(CCB)에서 승인/보류/거부 결정. 고객 PM 참여 필수
4
기준선 업데이트
승인된 변경사항을 요구사항정의서, Backlog, WBS, 테스트 케이스에 동시 반영
5
변경 이력 기록
변경 이력은 요구사항 버전 이력에 남기고 이전 버전은 보관합니다

변경 영향 체크

변경 요청이 승인되면 아래 항목을 모두 업데이트합니다:

  • 요구사항정의서 (버전 업)
  • 요구사항추적표
  • Backlog (Epic/Story/Task)
  • WBS (일정·담당자)
  • 설계 산출물 (영향 받는 것만)
  • 테스트 케이스 (AC 변경 시)

우선순위 재조정 시점

아래 상황에서 PM은 Backlog 우선순위 재조정을 고려합니다:

  • Sprint 중 신규 CR 승인
  • 고객 피드백으로 방향 변경
  • 인프라/환경 지연으로 Sprint 계획 변경
  • Must 요구사항 지연 발생
  • Phase Gate에서 결함 발견
